# Evaluation harness

This directory contains the benchmark used to check whether an embedding model
can find semantics-preserving rewrites before you trust its scan output.

## Planted-clone benchmark (`semdup inject-eval`)

Measures whether the embedding model can *find* a rewrite of a function it
has seen. We take real functions from public corpora, rewrite each at three
mutation levels, and check where the original ranks among the plant's
nearest neighbors:

- **L1 rename-only** — identical structure, every identifier and comment
  reworded. Tests robustness to vocabulary.
- **L2 restructured** — same algorithm, different control flow.
- **L3 re-derived** — written fresh from a one-paragraph spec of the
  original's behavior; only the *semantics* are shared.

Run it:

```bash
eval/fetch-corpus.sh
semdup extract --root eval/corpus --corpus main --granularity function
semdup extract --root eval/injected --corpus injected --granularity function
semdup embed
semdup inject-eval --manifest eval/manifest.json --min-recall5 0.9
```

For a candidate model, pass its cache key and ONNX export to `embed`, then
the same cache key to `inject-eval`:

```bash
semdup embed --model <MODEL> --model-dir <DIR>
semdup inject-eval --model <MODEL> --manifest eval/manifest.json
```

Report: per-plant cosine/rank/margin, then a per-level summary. A usable
model should hold recall@1 near 1.0 at L1 and keep the original in the top 5
for most L3 plants. **Margin** (cosine to original minus cosine to best
non-original) is the threshold-independent class-separation number to compare
across models.

`F1@1` is top-1 F1. Each plant has one relevant original and the top-1 result
makes one prediction, so precision@1 equals recall@1 and F1@1 has the same
value. This harness does not use F1@5 as a headline metric: with one relevant
item and five returned candidates, even perfect recall@5 has precision@5 =
0.20 and F1@5 = 0.33.

## Model matrix

Each published row is generated by one command. Known model aliases live in
`eval/models.tsv`; direct model ids also work when the default backend can
resolve them.

```bash
eval/model-row.sh --model nomic-ai/CodeRankEmbed --header
for model in fast-cpu fast-gpu; do
  eval/model-row.sh --model "$model"
done
```

Current published rows:

| model | n | L1 R@1 | L1 R@5 | L2 R@1 | L2 R@5 | L3 R@1 | L3 R@5 | macro F1@1 | L3 margin |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| nomic-ai/CodeRankEmbed | 84 | 0.89 | 1.00 | 0.86 | 0.96 | 0.75 | 0.96 | 0.83 | +0.0813 |
| CodeRankEmbed-nbits-int4-asym-cpu | 84 | 0.96 | 1.00 | 0.89 | 0.93 | 0.79 | 0.96 | 0.88 | +0.0683 |

The per-plant rows from `semdup inject-eval` are the audit trail for misses:
they show the known original's rank and the incorrect top hit when rank is
greater than 1.

Do not add a row to this table until it has been generated by
`eval/model-row.sh`; `semdup status` can show stale embeddings from old
corpus rows, but `model-row.sh` refreshes `eval/corpus` and `eval/injected`
before it evaluates.

## Block-granularity benchmark

`eval/block-manifest.json` is a smaller companion benchmark for inline-helper
detection. It evaluates executable block units instead of function units, and
manifest originals use line-spanned selectors such as
`crates/cli/src/pattern.rs:146-155::patterns_from_reader::match` so repeated
block names are unambiguous.

Run it:

```bash
eval/fetch-corpus.sh
semdup extract --root eval/corpus --corpus main \
  --granularity function --granularity block --min-block-lines 8
semdup extract --root eval/injected --corpus injected \
  --granularity function --granularity block --min-block-lines 8
semdup embed
semdup inject-eval \
  --manifest eval/block-manifest.json \
  --unit-kind block \
  --min-lines 8
```

The same row helper can run it:

```bash
eval/model-row.sh \
  --model nomic-ai/CodeRankEmbed \
  --manifest eval/block-manifest.json \
  --unit-kind block
```

## Quantized variants

The model matrix abstracts over two choices: model artifact and ONNX Runtime
execution provider. Known aliases in `eval/models.tsv` encode both choices,
so the user-facing backend remains `onnx`.

The current fast-path aliases are:

| alias | provider | artifact | local smoke result |
| --- | --- | --- | --- |
| `fast-cpu` | `cpu` | MatMulNBits int4 asymmetric ONNX | 0.58 texts/s on 104 injected units |
| `fast-gpu` | `cuda` | fp16 ONNX | 1.67 s on 101 injected units |

The default hosted model uses cache key `nomic-ai/CodeRankEmbed@fast`; it
picks MatMulNBits int4 on CPU and fp16 on CUDA. The fp16 graph is smaller,
but it is not a CPU speed path. For CPU experiments, start with the int4
quantization used by `fast-cpu`:

```bash
python3 scripts/quantize_onnx.py \
  --mode nbits-int4-asym \
  --input models/coderankembed-fp32 \
  --out models/coderankembed-nbits-int4-asym

eval/model-row.sh --model coderankembed-nbits-int4-asym-cpu
```

The generated int4 model is loadable by the built-in ONNX backend and was
about 149 MB locally versus 523 MB for fp32. On the full planted-clone eval it
improved macro F1@1 from 0.83 to 0.88 versus the fp32 CPU baseline. A small
CPU smoke on 104 injected units ran at 0.58 texts/s for int4 versus
0.34 texts/s for fp32; on the same smoke with an RTX 3080, int4 CUDA took
1.91 s, fp16 CUDA took 1.82 s, and fp32 CUDA took 2.47 s, so `fast-gpu` still
uses fp16.

FP8 is not currently wired into this repo. ONNX Runtime's documented CPU
quantization path is int8; the FP8 knobs in the public docs are for
specialized execution providers such as MIGraphX, not the CPU execution
provider used by semdup's default backend.

## Extending the benchmark

The plants live in `eval/injected/` with provenance headers;
`eval/manifest.json` maps each plant file to its original
(`path-suffix::name`) and level. `eval/block-manifest.json` uses
`path-suffix:start-end::name` for block originals. To extend: add a rewrite
file + manifest entry. Keep one primary function or block per file (the
largest matching unit kind in the file is taken as the plant), don't reuse the
original's name, and pick originals with real logic, not boilerplate.

## Picking a scan threshold

Thresholds are per repo and per model: different codebases have different
vocabulary overlap, and different models place the same pairs differently.
Sweep a few values on your own code and keep the one whose report you would
actually act on:

```bash
for t in 0.55 0.60 0.65 0.70 0.75; do
  semdup scan --threshold "$t" --skip-tests | tail -1
done
```

Then read the report at your candidate value. The boundary is between real
duplication and intentional parallelism such as read/write mirrors or
per-variant implementations, and only the repo owner can place it. Re-sweep
after switching models.
